# 사이트온 v3 종합 설계서

> 범위: 디자인 시스템 고도화 · 애니메이션 · 폰트 · 여백 · 섹션 아키텍처 · 영상 · 안정화 · 배포 · 워드프레스 연동
> 성격: 설계 문서 (구현 없음). 구현 단위로 바로 쪼갤 수 있는 수준의 구체성을 목표로 한다.
> 전제: 현재 v2 = 단일 HTML 빌더, 12스타일 × 11컬러, 토큰/블록/변형/인라인편집/보정/실행취소 보유.

---

## 0. 설계 원칙과 현재 진단

**원칙**
1. 모든 시각 결정은 토큰을 거친다 — 하드코딩된 색·크기·간격·시간 값은 기술 부채로 간주
2. 스타일(테마)은 "토큰 값 세트 + 섹션 변형 조합"으로 정의된다 — 제너레이터 분기 코드가 아니라 데이터
3. 미리보기 = 산출물. 에디터 전용 코드는 산출물에 단 1바이트도 남지 않는다
4. 한국 시장 우선: 한글 타이포 규칙, 네이버 생태계, 의료법, 카카오 연동이 1급 시민
5. 성능 예산을 디자인보다 위에 둔다 — 예산을 깨는 연출은 채택하지 않는다

**v2의 구조적 한계 (v3가 풀어야 할 것)**
- 12개 제너레이터가 마크업을 각자 소유 → 섹션 추가/수정 시 12곳 수정 (postProcess로 완화했지만 근본 해결 아님)
- base64 이미지 내장 → 사이트 5~10장 사진이면 파일 수 MB, localStorage 5MB 한계와 충돌
- 토큰이 1계층(즉시값) → 다크 스타일과 라이트 스타일에 같은 토큰을 안전하게 적용 불가
- 모션이 JS 산발 구현 → 타임라인 개념 부재, 섹션 간 연출 일관성 없음

## 1. 디자인 시스템 2.0 — 3계층 토큰 아키텍처

### 1.1 계층 구조

```
Primitive (원시값)        Semantic (의미)              Component (부품)
─────────────────        ──────────────────           ─────────────────
color.teal.500  ───────▶ color.accent        ───────▶ button.primary.bg
color.gray.900  ───────▶ color.text.strong   ───────▶ card.title.color
space.6 (=48px) ───────▶ space.section.inner ───────▶ hero.padding.bottom
duration.300    ───────▶ motion.enter        ───────▶ reveal.duration
```

- 스타일(테마)은 **Semantic 계층만** 재정의한다. Primitive는 전 스타일 공유, Component는 Semantic을 참조만 한다.
- 다크 스타일은 `color.text.strong → gray.50` 식으로 시맨틱 매핑만 뒤집는다. v2처럼 스타일마다 별도 CSS를 쓰는 일이 사라진다.

### 1.2 토큰 스키마 (JSON, 직렬화 가능 = 저장/마켓 유통 단위)

```json
{
  "$schema": "siteon/tokens-v3",
  "color":   { "accent": "{teal.500}", "accentDeep": "{teal.700}",
               "bg": "{white}", "bgAlt": "{gray.50}", "bgInverse": "{gray.950}",
               "text": {"strong": "{gray.900}", "body": "{gray.600}", "subtle": "{gray.400}", "onInverse": "{white}"},
               "line": {"default": "{gray.200}", "strong": "{gray.900}"} },
  "font":    { "display": "pretendard", "body": "pretendard",
               "scale": 1.25, "baseSize": 16,
               "tracking": {"display": -0.035, "body": -0.012, "label": 0.16},
               "leading":  {"display": 1.18, "body": 1.75} },
  "space":   { "unit": 8, "sectionY": "lg", "containerMax": 1160, "gutter": 24 },
  "radius":  { "scale": "soft" },
  "shadow":  { "level": 2 },
  "motion":  { "intensity": "standard", "speed": 1.0 }
}
```

- 컬러 11종·스타일 12종은 이 스키마의 **프리셋 인스턴스**가 된다. 신규 테마 = JSON 1개 추가.
- 사용자 커스텀(현 토큰 패널)은 이 JSON에 패치로 얹힌다 → 내보내기/가져오기/마켓 판매 모두 같은 형식.

### 1.3 컬러 설계

- 역할 토큰 12개로 고정: accent, accentDeep, accentSoft, bg, bgAlt, bgInverse, text.strong/body/subtle/onInverse, line.default/strong
- **자동 파생 알고리즘**: 사용자가 accent 1색만 고르면 OKLCH 색공간에서 파생 — accentDeep = L-18%, accentSoft = L+38% C-60%, bgInverse = accent의 H 유지 + L 12% C 10% (현재 hexMix의 단순 RGB 혼합보다 한국어 사이트의 탁한 파생색 문제를 해결)
- **대비 가드**: 토큰 조합 시 WCAG AA(4.5:1) 자동 검사. 미달이면 가장 가까운 통과값으로 스냅하고 에디터에 배지 표시
- 불투명도는 색 토큰의 속성이 아니라 **사용처 토큰**(예: `text.body.opacity`)으로 둔다 — v2에서 발견한 "본문만 흐리게" 요구의 일반화

### 1.4 폰트 설계

**페어링 프리셋 (한글 웹폰트 전략)**

| 프리셋 | Display | Body | 무드 | 로딩 전략 |
|---|---|---|---|---|
| 모던 | Pretendard Variable | Pretendard | 기본, 중립 | dynamic-subset (현행) |
| 클래식 | Noto Serif KR | Pretendard | 신뢰, 프리미엄 | 600/700만 서브셋 |
| 휴먼 | 학교안심/KoPub돋움 계열 | Pretendard | 친근, 소아 | woff2 self-host |
| 임팩트 | Paperlogy/Gmarket Sans | Pretendard | 볼드, 이벤트 | 디스플레이만 로드 |

- 서드파티 CDN 의존 제거: 배포 단계(6장)에서 woff2 self-host + `font-display: swap` + 서브셋(KS X 1001 + 자주 쓰는 확장 2,780자)
- **모듈러 스케일**: baseSize × scale^n. scale 선택지 1.2(차분)/1.25(기본)/1.333(드라마틱). 단계는 7개로 고정: caption, body, lead, h3, h2, h1, display
- **한글 자간-크기 연동 곡선**: 한글은 커질수록 자간을 좁혀야 한다. `tracking(size) = clamp(-0.05em, -0.012em - (size-16)*0.0011em, -0.012em)` — 토큰 패널의 자간 슬라이더는 이 곡선의 오프셋으로 동작 (절대값 강제 아님)
- 행간: display 1.12~1.25 / body 1.6~1.9 범위 가드. `word-break: keep-all` + `text-wrap: balance`(제목) 전역 기본

### 1.5 여백(스페이싱) 설계

- 8pt 그리드, 토큰 `s1(4)~s12(160)` 고정 사다리. 임의 px 금지
- **섹션 수직 리듬**: sectionY 토큰 3모드 — compact(s9=72), standard(s10=104), airy(s12=160). 내부 리듬은 비율로 종속: 라벨→제목 = sectionY×0.18, 제목→본문 = ×0.16, 헤더→콘텐츠 = ×0.5
- **컨테이너 시스템**: containerMax 3종(1160 기본 / 980 내로 / 1320 와이드) + 풀블리드. 섹션 변형이 컨테이너 폭을 선언적으로 선택
- 12컬럼 그리드, gutter 토큰(20/24/32). 비대칭 레이아웃은 "7+5", "8+4" 같은 컬럼 스팬 명세로 정의 — 현행 fr 하드코딩 대체

## 2. 섹션 아키텍처 — "데이터 + 변형 + 토큰 바인딩"

### 2.1 핵심 전환: 제너레이터 → 섹션 컴포저

v2: `genSoft()…genShowcase()` 12개 함수가 페이지 전체를 소유
v3: **섹션 렌더러 1세트 × 변형 × 토큰** 으로 역전

```
페이지 = compose(
  theme: tokens.json,
  layout: { order: [...], container: ... },
  sections: [ {type:'hero', variant:'video-center', data:{...}},
              {type:'services', variant:'alternating-5050', data:{...}}, ... ]
)
```

- "스타일 12종"은 (토큰 프리셋 + 섹션 변형 프리셋 + 순서 프리셋) 묶음으로 재정의 → 스타일은 코드가 아니라 **레시피 JSON**
- 레퍼런스 이식 파이프라인이 단순해진다: 실측 → 토큰값 + 변형 선택으로 기록

### 2.2 섹션 카탈로그 (타입 × 변형 매트릭스)

| 섹션 타입 | 변형 (각 3개 목표) | 비고 |
|---|---|---|
| hero | photo-left / video-center / split-5050 / typographic | 영상은 4장 참조 |
| about | editorial(원칙 리스트) / statement(큰 인용) / timeline(연혁) | |
| services | card-grid / numbered-list / alternating-5050 / accordion / h-scroll | v2 자산 흡수 |
| doctors | card / list / interview(1인 큰 사진+Q&A) | interview 신규 |
| equipment | spec-grid(장비 사진+스펙) / badge-wall | 신규 |
| gallery | grid / masonry / filmstrip | |
| stats | strip / inline-rule / big-numbers | |
| process | step-timeline / vertical-flow | v2 process 스타일에서 추출 |
| faq | accordion / two-column | FAQPage JSON-LD 유지 |
| pricing | table / cards | 의료법 단서문구 강제 포함 |
| transport | rows / map-side | |
| event | banner / countdown | |
| booking | inline-strip / full-form / cta-only | 기본 off 유지 |
| location | split+map / full-map | |
| custom-html | raw | 파워유저용, 새니타이즈 필수 |

- 신규 섹션 추가 비용 = 렌더러 1개 + 스키마 1개. 12배 비용 구조 종결.

### 2.3 섹션 데이터 스키마

각 섹션은 `{type, variant, visible, container, data, overrides}` 5필드. `overrides`는 해당 섹션 한정 토큰 패치(예: 이 섹션만 bgInverse). 인라인 편집 레지스트리는 이 스키마에서 **자동 생성**된다 — v2의 텍스트 값 매칭(취약)을 경로 기반 바인딩(`sections[3].data.items[0].title`)으로 교체.

## 3. 애니메이션·모션 설계

### 3.1 모션 토큰

```
duration: xs 120ms / sm 240 / md 420 / lg 700 / xl 1100   (×speed 계수)
easing:   enter  cubic-bezier(.16,1,.3,1)    — 감속, 등장 전용
          exit   cubic-bezier(.7,0,.84,0)    — 가속, 퇴장 전용
          move   cubic-bezier(.22,1,.36,1)   — 위치 이동
          spring cubic-bezier(.34,1.3,.64,1) — 팝(프렌들리 계열만 허용)
distance: reveal-y 24/32/44px (intensity 연동)
```

intensity 3단계(calm/standard/expressive)가 duration·distance·연출 개수를 일괄 스케일 — 현 모션 프리셋 4종을 흡수.

### 3.2 오케스트레이션 규칙 (섹션 입장 타임라인)

섹션이 뷰포트에 들어올 때 내부 요소는 항상 같은 문법으로 등장한다:

```
t=0        라벨(레이블)        fade + y8
t=+90ms    제목               fade + y(reveal-y), 단어 스태거 60ms(히어로만)
t=+180ms   본문/설명           fade + y(reveal-y×0.6)
t=+260ms   미디어/카드 그룹     fade + y, 아이템 스태거 70ms (최대 6개까지만, 이후 동시)
```

- 스태거 상한(6개) — v2에서 카드 8개 스타일의 늘어지는 꼬리 제거
- 동일 뷰포트에 두 섹션이 동시에 들어오면 두 번째 섹션은 타임라인을 200ms 지연 (충돌 방지)

### 3.3 스크롤 연동 설계

- **1차: CSS Scroll-Driven Animations** (`animation-timeline: view()`) — 패럴랙스·진행바·이미지 리빌을 JS 없이 컴포지터에서 처리. 미지원 브라우저는 **2차: IO+rAF 폴백**(현행 코드 재사용)
- 패럴랙스 깊이 3레이어 규약: 배경 ±28px / 미디어 ±16px / 전경 장식 ±40px. 텍스트는 패럴랙스 금지(가독성)
- 스크롤 진행바, 히어로 페이드아웃, 스티키 내비 축소는 표준 연출로 전 테마 공통
- 신규 연출 후보(intensity=expressive에서만): 섹션 배경색 크로스페이드(다크↔라이트 섹션 경계), 숫자 카운터 + 게이지 라인, 갤러리 필름스트립 가로 스크럽

### 3.4 마이크로 인터랙션 카탈로그

| 대상 | 기본 | 호버 | 액티브 |
|---|---|---|---|
| 버튼 primary | — | bg→accentDeep, y-2px, duration.sm | scale .98 |
| 카드 | — | y-6px + shadow level+1, duration.md/move | — |
| 텍스트 링크 | — | 밑줄 좌→우 드로우 240ms | — |
| 입력 필드 | — | border accent | focus ring 4px accentSoft |
| 이미지 | reveal시 scale 1.06→1 | scale 1.04 (컨테이너 클립) | — |

### 3.5 성능 가드 (모션 헌법)

1. transform/opacity 외 속성 애니메이션 금지 (height/box-shadow 트랜지션은 컴포지터 비용 검토 후 예외 승인)
2. `will-change`는 애니메이션 직전 부여, 종료 후 제거 — 상시 선언 금지
3. 스크롤 핸들러는 단일 rAF 디스패처 1개로 통합 (v2는 5개 산재 → 통합)
4. `prefers-reduced-motion: reduce` → 모든 모션 토큰 0 (현행 유지)
5. 60fps 예산: 한 프레임에 움직이는 요소 12개 상한

## 4. 영상 설계

### 4.1 히어로 영상 파이프라인 (현 유튜브 임베드의 한계와 로드맵)

| 단계 | 방식 | 장점 | 한계 |
|---|---|---|---|
| 현행 | YouTube iframe 무음 루프 | 무료, 구현 쉬움 | LCP 차단, 광고/추천 리스크, 모바일 자동재생 불안정, 쿠키 |
| v3-a | youtube-nocookie + facade 패턴 | 쿠키/성능 개선 | 근본 한계 동일 |
| v3-b (목표) | 자체 호스팅 mp4/webm + CDN | 완전 제어, 빠름 | 스토리지/인코딩 파이프라인 필요 (6장) |

**v3-b 명세**
- 업로드 → 서버 인코딩: H.264 mp4(호환) + VP9/AV1 webm(용량), 720p/1080p 2렌디션, 10초 내외 루프 권장 가이드 UI
- `<video autoplay muted loop playsinline preload="metadata" poster>` — poster는 첫 프레임 자동 추출 webp
- **LCP 보호**: poster 이미지를 LCP 요소로 삼고 영상은 `requestIdleCallback` 후 attach. 영상이 LCP가 되는 구조 금지
- **대역폭 정책**: `(prefers-reduced-data)` 또는 모바일+save-data → poster 정지 이미지로 대체. 뷰포트 이탈 시 `video.pause()`
- 오버레이는 토큰화: `hero.overlay.opacity` (현 고정 0.45 → 사용자 조절)

### 4.2 본문 영상

- 소개 영상 섹션: 현행 유튜브 유지하되 **facade**(썸네일+재생버튼, 클릭 시 iframe 주입)로 초기 로드 0KB
- 갤러리에 짧은 클립(시설 투어) 허용: 라이트박스에서 video 태그 재생

## 5. 안정화 설계

### 5.1 테스트 전략 (4층)

| 층 | 도구 | 대상 | 게이트 |
|---|---|---|---|
| 단위 | Vitest | 토큰 파생, esc, 스키마 검증, postProcess | PR 필수 |
| 골든(스냅샷) | 자체 러너 | 12스타일(→레시피) × 대표 변형 출력 HTML 해시 비교 | PR 필수 |
| E2E | Playwright | 시나리오 10: 편집→미리보기 동기화, 인라인 편집, 업로드→보정→적용, undo, 저장/복원, 내보내기 유효성, 모바일 메뉴, 폼, 순서변경, 토큰 | nightly |
| 시각 회귀 | Playwright 스크린샷 diff | 스타일×3뷰포트×주요 섹션 | 릴리스 전 |

- v2 QA에서 배운 것 반영: 백그라운드 탭 IO/rAF 정지 → 테스트 러너는 headed+포커스 보장 또는 IO 모킹

### 5.2 성능 예산 (산출물 기준)

- LCP < 2.5s(4G 모토G급), CLS < 0.1, INP < 200ms, 초기 HTML ≤ 80KB(이미지 제외), JS ≤ 12KB
- 이미지: 페이지 합계 ≤ 1.2MB, AVIF/WebP 우선, `srcset` 3브레이크포인트, LQIP 블러 플레이스홀더
- **base64 탈피 로드맵**: 빌더(로컬) 단계에서는 IndexedDB로 원본 보관 + 내보내기 시 `/assets` 폴더 zip 옵션, SaaS 단계에서는 스토리지 URL (6장)

### 5.3 데이터 안정성

- localStorage(5MB) → **IndexedDB** 이행: 프로젝트 메타는 LS 유지, 이미지 blob은 IDB. 쿼터 초과 시 사용자 경고 + 오래된 히스토리 자동 정리
- 히스토리: 전체 스냅샷(현행) → **diff 패치 스택**으로 교체 (메모리 60×수MB 문제 해결)
- 스키마 버전 필드 + 마이그레이터 체인 (v1→v2→v3 프로젝트 파일 자동 승급) — 기존 사용자 파일 호환 보장
- 충돌 방지: 같은 프로젝트 다중 탭 편집 감지(BroadcastChannel) → 후발 탭 읽기전용

### 5.4 보안·접근성·브라우저

- 새니타이즈: esc() 전수 유지 + custom-html 섹션은 DOMPurify 도입. 내보내기 HTML에 CSP meta 권장값 주석
- 접근성 AA 체크리스트: 포커스 트랩(모바일 메뉴/라이트박스/보정 모달), 대비 자동검사(1.3절), 모든 인터랙티브 요소 키보드 도달, 시맨틱 랜드마크(header/main/footer), 이미지 alt 강제 입력 UI
- 지원 매트릭스: Chrome/Edge/Safari/삼성인터넷 최근 2버전 + iOS Safari 16+. `color-mix`·`:has` 등은 폴백 명세 동반

## 6. 배포 아키텍처 (SaaS 전환)

### 6.1 단계별 전략

```
Phase 1 (현행+)      Phase 2 (호스팅 MVP)              Phase 3 (스케일)
정적 HTML 내보내기 → 원클릭 퍼블리시 + 서브도메인     → 커스텀 도메인/팀/마켓
```

### 6.2 스택과 토폴로지

- **앱**: Next.js(App Router) — 에디터(/studio)는 CSR, 게시 사이트는 SSG
- **데이터**: Supabase — Auth(카카오/이메일), Postgres, Storage(이미지/영상), Row Level Security
- **호스팅/CDN**: Vercel(앱) + Cloudflare(게시 사이트 캐시, 와일드카드 도메인 `*.siteon.kr`)
- **퍼블리시 파이프라인**: 저장 → 사이트 JSON(2장 스키마) → 렌더 함수(빌더와 동일 코드를 서버에서 실행) → 정적 HTML/자산 → Storage/CDN 업로드 → 캐시 무효화. 평균 게시 시간 목표 < 10초
- **커스텀 도메인**: 고객 CNAME → Cloudflare for SaaS(SSL 자동 발급) → 도메인-사이트 매핑 테이블

### 6.3 DB 스키마 (요지)

```
users(id, email, kakao_id, plan, created_at)
sites(id, owner_id, slug, custom_domain, recipe_jsonb, tokens_jsonb, status, published_at)
site_versions(id, site_id, snapshot_jsonb, created_at)        -- 게시 이력/롤백
assets(id, site_id, kind[image|video], storage_path, variants_jsonb, bytes)
leads(id, site_id, name, phone, service, message, source, created_at)  -- 예약/문의
subscriptions(id, user_id, toss_billing_key, plan, status, renews_at)
```

### 6.4 폼·알림 (예약 리드 백엔드)

- 게시 사이트 폼 → Edge Function → leads INSERT → 알림 팬아웃: 카카오 알림톡(1순위, 솔라피/알리고 게이트웨이) + 이메일 폴백 + 빌더 대시보드 리드함
- 스팸: rate limit(IP+사이트), honeypot 필드, 전화번호 형식 검증

### 6.5 운영

- 모니터링: Sentry(에디터/렌더러), 업타임 체크(게시 도메인 샘플링), Web Vitals RUM 수집(게시 사이트 1% 샘플)
- 백업: Postgres PITR + site_versions 30일 / 자산은 Storage 버저닝
- 결제: 토스페이먼츠 빌링 — 플랜 게이트(사이트 수, 커스텀 도메인, 영상 호스팅 용량)

## 7. 워드프레스 연동 설계

### 7.1 전략 3안 비교

| 안 | 구조 | 장점 | 단점 | 판정 |
|---|---|---|---|---|
| A. 테마 익스포트 | 사이트온 → WP **블록 테마 zip** (theme.json + 블록 패턴) | WP 생태계 완전 편입, 고객이 WP에서 계속 편집 | 사이트온과 양방향 동기화 불가 | 채택(보조) |
| B. 임포터 플러그인 | WP 플러그인이 사이트온 프로젝트 JSON을 받아 Gutenberg 콘텐츠로 변환 | 기존 WP 사이트에 섹션 단위 이식 가능 | 변환 충실도 한계 | 채택(주력) |
| C. 헤드리스 | WP REST를 콘텐츠 소스로, 프런트는 사이트온 렌더러 | 디자인 100% 보존 | 고객사 인프라 요구 높음, 시장성 낮음 | 보류 |

**권장: B(주력) + A(보조).** 한국 병원 시장의 실수요는 "기존 WP/그누보드 사이트를 사이트온 디자인으로 교체"와 "원장님이 익숙한 WP 관리자에서 글만 쓰고 싶다" 두 가지 — B가 전자, A가 후자를 커버.

### 7.2 매핑 명세

**토큰 → theme.json** (A안 핵심)

```
color.accent        → settings.color.palette[primary]
font.display/scale  → settings.typography.fontFamilies / fontSizes (모듈러 스케일 그대로)
space.unit/sectionY → settings.spacing.spacingSizes
radius / shadow     → styles.blocks 공통 변수(custom)
```

**섹션 → 블록 패턴** (양안 공통)

| 사이트온 섹션 | WP 구성 |
|---|---|
| hero | cover 블록(+video) + heading + buttons 패턴 |
| services | columns × (image+heading+paragraph) 패턴, accordion 변형은 details 블록 |
| doctors | query loop ← CPT `doctor` (ACF: 직책/이력/사진) |
| faq | details 블록 반복 + FAQPage JSON-LD는 플러그인이 head 주입 |
| pricing/transport | table / columns 패턴 |
| booking | WPForms/Contact Form 7 매핑 또는 자체 폼 블록 |
| location | 커스텀 map 블록(주소 attr) |

**CPT/ACF 설계** (B안): `doctor`, `service`, `faq` 3개 커스텀 포스트 타입 — 콘텐츠가 데이터로 살아남아 WP쪽에서 추가/수정 가능. 사이트온 JSON ↔ CPT 직렬화 스펙 1:1.

**모션의 한계 처리**: WP 변환 시 모션은 "기본 리빌 + 마이크로 인터랙션"까지만 보존(경량 JS 1파일 동봉). 스크롤 연출 고급 모드는 사이트온 호스팅 전용 차별점으로 남긴다 — 의도된 비대칭.

### 7.3 산출물

- `siteon-importer.zip` (플러그인): JSON 업로드 UI, 미디어 사이드로드(base64→미디어 라이브러리), 패턴 등록, CPT 마이그레이션, 진행률/롤백
- `siteon-theme-{스타일}.zip` (블록 테마): theme.json + templates + patterns + style.css(토큰 변수)

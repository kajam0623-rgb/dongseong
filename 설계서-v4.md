# 사이트온 v4 설계서 — 듀얼 모드: 원페이지 ↔ 멀티페이지

> 전제: `설계서-v3.md`(토큰 3계층 · 섹션 컴포저 · 모션 토큰 · 배포 · WP 연동)를 상속한다. v4는 그 위에 **사이트 모드** 차원을 추가한다.
> 성격: 설계 문서. 구현 없음.

---

## 1. 모드 개념

```
siteMode: 'onepage' | 'multipage'
```

- **원페이지**: 현행 구조. 모든 섹션이 한 화면, 내비는 앵커 스크롤. 개원 초기·예산 최소·빠른 오픈에 적합
- **멀티페이지**: 홈 + 세부페이지 트리. 진료과목별 상세, 의료진 상세, 칼럼 등. SEO 성장·중대형 의원에 적합
- 선택 지점 2곳: **온보딩 위저드 1번 질문** + 에디터 설정에서 **언제든 비파괴 전환**(3장 변환 규칙). 전환은 미리보기를 먼저 보여주고 확정한다 — 되돌리기는 히스토리로 보장
- 마케팅 포지션: 원페이지 = 베이직 플랜, 멀티페이지 = 프로 플랜 (요금 차별화 축)

설계 핵심 명제: **두 모드는 다른 제품이 아니다.** 같은 컬렉션 데이터(5장)와 같은 섹션 컴포저를 쓰고, 차이는 "섹션을 한 문서에 펼치느냐, 페이지 트리에 분배하느냐"뿐이다. 이 명제가 지켜지면 전환·유지보수·WP 연동이 전부 단순해진다.

## 2. 정보 구조(IA) — 치과 멀티페이지 표준 사이트맵

```
/                     홈 (다이제스트: 히어로 + 각 영역 요약 + CTA)
/about                치과소개 (인사말 · 진료철학 · 둘러보기)
/doctors              의료진 목록
/doctors/{slug}       의료진 상세 (프로필 · 약력 · 담당진료 · 칼럼)
/services             진료과목 허브
/services/{slug}      진료 상세 ★SEO 주력 (임플란트/보철/라미네이트/교정…)
/facility             시설 · 장비
/cases                치료 사례 (의료광고 심의 통과 시 옵트인, 기본 숨김)
/columns              칼럼 목록 (SEO 콘텐츠 엔진)
/columns/{slug}       칼럼 상세
/notice               공지 · 이벤트
/faq                  자주 묻는 질문
/nonpayment           비급여 진료비 고지 (의료법 45조 — 게시 게이트)
/privacy              개인정보처리방침 (자동 생성)
/contact              오시는길 · 진료시간 · 예약
```

- 위 트리는 **기본 프리셋**이며 페이지 추가/삭제/숨김 자유. 단 `/nonpayment`, `/privacy`는 삭제 불가(숨김만 가능, 게시 게이트 경고)
- 깊이는 2단계로 제한(허브/상세). 3단계 이상은 의원급 사이트에서 이탈만 늘린다 — 의도적 제약

## 3. 모드 변환 규칙 (비파괴 양방향)

### 3.1 원페이지 → 멀티페이지 (승격)

| 원페이지 섹션 | 생성되는 페이지 | 규칙 |
|---|---|---|
| hero | / 홈 히어로 | 그대로 |
| about | /about | about 섹션 + 갤러리 일부 이동, 홈에는 3줄 요약 + 링크 카드 잔류 |
| services | /services 허브 + 항목당 /services/{slug} | 각 항목이 상세 페이지 초안으로 승격(4.2 레시피 적용), 홈에는 카드 그리드 잔류(링크가 앵커→URL로 전환) |
| doctors | /doctors + 인원당 상세 | 동일 패턴 |
| gallery | /facility | 홈에 대표 3장 잔류 |
| faq | /faq | 홈에 톱3 잔류 + 더보기 링크 |
| pricing 블록 | /nonpayment | 비급여 고지 공식 페이지로 승격 |
| stats·event·booking·location | 홈 잔류 | contact 페이지에 location 복제 배치 |

- 승격은 **복사가 아니라 참조 재배치**: 컬렉션(5장) 데이터는 한 곳, 페이지는 뷰만 소유 → 전환 후에도 수정은 한 곳에서
- 상세 페이지 초안의 빈 필드(개요·프로세스 등)는 placeholder + AI 카피라이터 연결 지점으로 표시

### 3.2 멀티페이지 → 원페이지 (다이제스트)

- 각 페이지를 "대표 섹션 1개"로 축약해 순서대로 적층: 상세 본문은 잃지 않고 **보관 상태**로 유지(데이터 삭제 없음) — 다시 멀티로 돌리면 복원
- 축약 규칙: services 상세 → 카드 1장(제목+요약+사진), columns → 최신 3개 리스트 섹션, doctors 상세 → 카드

### 3.3 전환 UX

전환 다이얼로그: 좌(현재) / 우(전환 후) 미리보기 → "전환 후 생성될 페이지 N개 / 잔류 섹션 목록" 요약 → 확정. 확정 전 어떤 데이터도 변형하지 않는다.

## 4. 페이지 템플릿(레시피) 카탈로그

각 페이지는 섹션 컴포저 레시피로 정의된다. 기본 레시피는 아래와 같고, 사용자는 섹션을 자유 편집한다.

### 4.1 홈 (멀티 모드)

`hero → trust-bar(통계·인증 한 줄) → services 카드(상세 링크) → doctors 미리보기 → facility 대표 3장 → columns 최신 3 → faq 톱3 → CTA → location 요약`

홈의 역할이 "모든 정보"에서 "각 페이지로의 분배기"로 바뀐다 — 섹션당 분량을 원페이지의 60% 수준으로 압축하는 것이 기본값.

### 4.2 진료 상세 `/services/{slug}` ★ 가장 중요한 페이지

```
sub-hero        소제목 히어로(페이지명 + 1줄 가치제안 + 브레드크럼) — 풀히어로의 55% 높이
overview        이 치료가 필요한 경우 (적응증 리스트, keep-all 불릿)
process         치료 과정 step-timeline (v3 process 섹션 재사용)
equipment       사용 장비 (장비 컬렉션에서 태그 필터)
pricing-ref     비용 안내 — /nonpayment 해당 항목 자동 인용 + 단서문구
doctors-ref     담당 의료진 (doctors 컬렉션에서 담당진료 매칭)
faq-ref         관련 FAQ (faq 컬렉션 태그 필터)
case-ref        관련 사례 (옵트인 시)
cta             전화 + 네이버 예약
```

`-ref` 섹션들은 **컬렉션 참조 섹션**: 데이터를 복제하지 않고 쿼리(태그/관계)로 표시 — 진료과목이 10개라도 FAQ는 한 곳에서 관리.

### 4.3 기타 페이지 기본 레시피

| 페이지 | 레시피 |
|---|---|
| /about | sub-hero → 인사말(원장 사진+서명) → 진료철학 3원칙 → 연혁 timeline → 둘러보기 갤러리 → CTA |
| /doctors/{slug} | sub-hero(프로필형: 사진 좌, 이름·직책 우) → 약력 리스트 → 담당진료 카드(services-ref) → 칼럼(columns-ref author 필터) → CTA |
| /columns/{slug} | 글 히어로(제목·날짜·작성 의료진) → 본문(rich text) → 관련 진료 카드 → CTA. Article JSON-LD |
| /contact | sub-hero → 진료시간 표 → 오시는길+지도 풀와이드 → 교통편 → 예약 폼(옵션) → 퀵 연락 카드 |
| /nonpayment | 법정 양식 표(고시 항목명·금액·비고) + 게시일 자동 표기 |

### 4.4 sub-hero 신규 변형

세부페이지 공용 헤더. 변형 3종: bar(컴팩트 띠) / photo(배경사진 55vh) / clean(타이포만). 브레드크럼 내장(BreadcrumbList JSON-LD 자동).

## 5. 데이터 스키마 v4 — 컬렉션 중심 재편

```json
{
  "$schema": "siteon/site-v4",
  "mode": "multipage",
  "globals": { "identity": {"name","logo","phone","address","hours","naverUrl","kakaoChannel"},
               "tokens": { ... v3 토큰 ... },
               "nav": {"auto": true, "items": []},
               "compliance": {"nonpaymentItems": [], "adReviewNo": "", "privacyMeta": {}} },
  "collections": {
    "services": [ {"slug","title","summary","body","image","tags","faqTags","doctorIds"} ],
    "doctors":  [ {"slug","name","role","bio[]","photo","serviceIds"} ],
    "faq":      [ {"q","a","tags"} ],
    "posts":    [ {"slug","title","body","authorId","publishedAt"} ],
    "assets":   [ {"id","kind","src|storagePath","alt","meta"} ]
  },
  "pages": [ {"slug":"/", "title","seo":{...}, "sections":[ {type, variant, dataRef|data, overrides} ], "locales":{}} ]
}
```

핵심 결정:
1. **컬렉션이 단일 진실** — 페이지 섹션은 `dataRef`(컬렉션 쿼리)나 인라인 `data` 중 하나. 원페이지/멀티 전환은 pages만 재구성하면 끝
2. 이미지가 `assets` 컬렉션으로 분리 — base64 탈피(v3 5.3)와 정합
3. WP 연동(v3 7장)이 공짜로 깔끔해짐: collections.services → CPT `service`, posts → WP post — 1:1
4. v2/v3 프로젝트 파일은 마이그레이터가 `mode:'onepage'` + 컬렉션 추출로 자동 승급

## 6. 내비게이션 설계

| 요소 | 원페이지 | 멀티페이지 |
|---|---|---|
| 헤더 메뉴 | 앵커(#services…) | URL + 진료과목 드롭다운(1단), 현재 페이지 강조 |
| 모바일 메뉴 | 1단 리스트 | 아코디언 2단(허브 탭→하위), 하단 전화·네이버 고정 |
| 브레드크럼 | 없음 | sub-hero 내장, 홈 제외 전 페이지 |
| 푸터 | 연락처 1단 | 사이트맵 컬럼형(페이지 트리 자동) + 법정 링크(비급여·개인정보) |
| 퀵바 | 현행 | 동일 + 채팅 위젯 슬롯 (전 페이지 고정) |

- nav.auto=true면 페이지 트리에서 메뉴 자동 생성(숨김 페이지 제외), 수동 모드에서 순서·라벨 커스텀
- 헤더/푸터/퀵바는 **글로벌 파셜**: 한 번 렌더 정의, 전 페이지 공유. 페이지별 오버라이드는 투명 헤더 여부 정도만 허용

## 7. URL · 내보내기 · 라우팅

- **슬러그 규칙**: 영문 소문자-하이픈 기본(`/services/implant`). 한글 슬러그 옵션 제공하되 기본값은 영문(공유·인코딩 문제 회피). 자동 변환표: 임플란트→implant, 라미네이트→laminate, 교정→ortho, 보철→prosthetics 등 치과 용어 사전 내장
- **정적 내보내기(zip)**:
```
site.zip
├─ index.html
├─ about/index.html
├─ services/index.html
├─ services/implant/index.html …
├─ assets/site.css      ← 공통 CSS 추출(멀티 모드 전용, 페이지당 중복 제거)
├─ assets/site.js       ← 공통 인터랙션
├─ assets/img/…         ← base64 → 파일 분리
├─ sitemap.xml / robots.txt
```
  원페이지 모드는 현행 단일 html 유지(차별점: "파일 1개" 단순함을 잃지 않는다)
- **빌더 미리보기 라우팅**: iframe 내 링크 클릭 인터셉트 → 해당 페이지 레시피로 재렌더(가상 라우팅) + 에디터 상단에 페이지 탭/주소 표시. 뒤로가기는 에디터가 페이지 스택 관리
- **SaaS 라우팅**: pages 트리 → Next.js 동적 라우트, ISR 페이지 단위 재생성(수정된 페이지만 10초 내 재게시)

## 8. SEO 멀티페이지 전략

- **메타 자동 생성 템플릿**: 진료 상세 = `{지역} {진료명} | {병원명}` / desc = summary 앞 140자. 수동 오버라이드 가능, 비어 있으면 템플릿
- **구조화 데이터 페이지 타입별**: 홈 Dentist(현행) / 진료 상세 MedicalWebPage+MedicalProcedure / 의료진 Physician / 칼럼 Article / FAQ FAQPage / 전 페이지 BreadcrumbList
- **내부 링크 규칙(자동)**: 진료 상세 ↔ 관련 칼럼 ↔ 관련 FAQ 상호 링크는 컬렉션 태그에서 자동 — "고아 페이지 0" 보장
- sitemap.xml(우선순위: 홈 1.0, 진료 상세 0.9, 칼럼 0.7), robots.txt, 네이버 서치어드바이저 인증 메타 슬롯
- **칼럼 엔진이 멀티 모드의 존재 이유**: 주 1회 칼럼 → 진료 상세로 내부링크 → 지역+진료 키워드 장악. 온보딩에서 칼럼 주제 12개 추천 리스트 제공(예: "임플란트 후 음식", "라미네이트 수명")

## 9. 에디터 UX 변경

- 사이드바 최상단에 **페이지 패널** 신설: 트리 뷰(홈 고정 + 페이지 목록), 추가(템플릿 선택)/복제/숨김/삭제, 드래그 순서(=메뉴 순서)
- 페이지 선택 → 기존 섹션 패널이 해당 페이지 스코프로 전환 (UI 재사용, 학습 비용 0)
- 글로벌 항목(기본정보·테마·토큰·의료진·진료과목 컬렉션)은 "사이트 전체" 탭으로 분리 — 페이지 편집과 시각적으로 구분
- 인라인 편집·실행취소·사진보정은 페이지 스코프로 그대로 동작. 히스토리 스택은 사이트 단위(페이지 간 이동도 undo 가능)
- 모드 뱃지를 상단바에 상시 표시: `원페이지` / `멀티 · 14p`

## 10. 성능 · 안정화 보완 (멀티 특화)

- 예산은 **페이지 단위**로 동일 적용(v3 5.2). 공통 CSS/JS 추출로 2페이지째부터 캐시 히트
- 페이지 간 전환 연출: View Transitions API(같은 헤더 morph, 지원 브라우저 한정) — intensity=expressive에서만
- 골든 테스트 확장: 모드 2 × 대표 페이지 5종 스냅샷 추가
- E2E 추가 시나리오: 모드 전환 왕복 무손실, 허브→상세 링크 정합, zip 구조 검증, 비급여 게이트 차단

## 11. 로드맵 수정 (v3 S1~S7에 삽입)

| 스프린트 | 내용 |
|---|---|
| S2 (수정) | 섹션 컴포저 + **스키마 v4(컬렉션·pages·mode) 동시 확정** — 스키마 두 번 깨지 않는다 |
| **S2.5 (신설)** | 페이지 시스템: 페이지 패널, 가상 라우팅 미리보기, sub-hero, zip 내보내기, 모드 전환(원→멀티 단방향 먼저) |
| S3~S5 | v3 동일 (모션/미디어/안정화) — 멀티 페이지 골든 포함 |
